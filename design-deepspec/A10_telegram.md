# A10 · Telegram-бот — `@xerocode_bot`

> **Глубокая дизайн-спецификация одного экрана XeroCode v3.**
> Источник правды по мокапу: `XEROCODE_FULL.html` строки 2610–2655 (SCREEN 10).
> Источник правды по стилям веб-превью: `Enhance Office View Design/src/styles/xerocode-v3.css` строки 1471–1547 (`.tg-frame` → `.tg-input .send`).
> Бэкенд (реальный, уже существует): `backend/app/api/routes/telegram.py`, `backend/app/api/routes/oauth.py` (Telegram Login Widget), `backend/app/models/integration.py` (`Integration`, `TelegramLink`).
>
> ⚠️ **Двойная природа экрана.** Это НЕ React-экран нашей дизайн-системы. Это **превью** Telegram-чата внутри лендинга/брендбука (`browser-frame` → `tg-frame`), который имитирует реальный мессенджер. Поэтому спецификация делится на два пласта:
> 1. **Веб-превью (наш контроль)** — токены, шрифты, CSS-классы `.tg-*`. Здесь действует дизайн-система XeroCode.
> 2. **Реальный бот в Telegram (вне нашего контроля)** — нативные элементы мессенджера (системные пузыри, reply-keyboard, inline-кнопки, голос, скрепка). Здесь мы НЕ рисуем UI — мы только формируем **контент** (текст, разметку, набор кнопок). Внешний вид задаёт клиент Telegram (iOS/Android/Desktop) и тема пользователя.

## Назначение

**A10 — мобильный канал входа в XeroCode без установки приложения.** Пользователь открывает `t.me/xerocode_bot`, привязывает аккаунт XeroCode и получает «карманный AI-офис»: диалог с моделями, быстрые команды по своим данным (CRM, задачи, статус команды), генерация follow-up'ов и текстов «на ходу», итерация тона прямо в переписке.

Роль в продукте (по брифу §8, A10):
- **Точка входа с нулевым трением** — не нужен браузер, логин-форма, установка. Достаточно Telegram, который уже стоит у целевой аудитории (solo + SMB до 50 seats, РФ-рынок).
- **«Долинейный» канал** — короткие задачи, которые быстрее сделать в переписке, чем открывать веб: «напиши follow-up», «дай топ-5 сделок», «статус команды».
- **Демо-витрина продукта** на лендинге — экран A10 показывает живой сценарий (follow-up клиенту + итерация тона «сделай посерьёзнее»), доказывая ценность «один человек делает работу пятерых».

**Что НЕ является назначением:** бот — не замена веб-офису. Тяжёлые воркспейсы (Оркестрация/DAG, Код-редактор с diff, Документы-превью на Cream, batch-картинки) живут только в вебе. Граница описана в разделе «Граница бот vs веб».

## Доступ (привязка аккаунта XeroCode)

Бот публичный (`/start` доступен любому), но **полезные действия по данным организации требуют привязки** Telegram-аккаунта к аккаунту XeroCode. В коде это таблица `telegram_links` (`backend/app/models/integration.py`): `user_id ↔ telegram_user_id (unique) + telegram_username + linked_at`.

Три способа привязки (1 реализован, 2-3 — **ПРЕДЛОЖЕНИЕ**):

1. **Telegram Login Widget → веб (РЕАЛИЗОВАНО).**
   В вебе на A01 (Авторизация) есть OAuth-кнопка `Telegram`. Бэкенд `oauth.py · POST /api/auth/oauth/telegram/verify` проверяет HMAC-подпись payload'а виджета (`secret = sha256(bot_token)`, `hmac.new(secret, data_check_string)`), синтезирует `email = tg-{tg_id}@telegram.local`, заводит/находит юзера и выдаёт JWT. Это вход в веб через Telegram — и одновременно естественный момент для записи `TelegramLink`.

2. **Deep-link токен из веба (ПРЕДЛОЖЕНИЕ, рекомендуется как основной для бота).**
   В вебе (A09 Настройки → Интеграции → «Telegram-бот») кнопка **«Привязать Telegram»** генерирует одноразовый токен и показывает:
   - **Кнопку-ссылку** `Открыть @xerocode_bot →` вида `https://t.me/xerocode_bot?start=link_<token>` (Telegram прокидывает `<token>` первым аргументом в `/start`).
   - **QR-код** того же deep-link (для перехода с десктоп-веба на телефон).
   Пользователь жмёт `/start`, бот видит `link_<token>`, валидирует, создаёт `TelegramLink`, отвечает «✓ Аккаунт XeroCode привязан». Механика 1-в-1 повторяет device-code из `cli_auth.py` (`_pending`/`_by_user_code`, TTL, GC) — переиспользуем паттерн, а не пишем заново.

3. **Привязка изнутри бота по коду (ПРЕДЛОЖЕНИЕ, fallback).**
   `/start` без токена → бот отвечает «Чтобы привязать аккаунт, введите код из Настроек XeroCode (Интеграции → Telegram)». Юзер копирует короткий код `ABCD-EFGH` из веба, шлёт в бот, бот матчит и привязывает. Зеркально device-flow CLI (`user_code`).

**Текущее состояние кода:** `telegram.py · /start` сейчас отвечает заглушкой («Привет! Я XeroCode Bot. Используйте /task, /status») и **не реализует саму привязку** — есть только чтение `TelegramLink` для команд. Запись связи — открытая задача (см. «Открытые вопросы»).

**Уровень привязки.** `TelegramLink → user → organization_id`. Все командные ответы фильтруются по `organization_id` (правило проекта «Org isolation»): `/deals` и `/status` в коде уже джойнят `User.organization_id`. Один Telegram-аккаунт = один XeroCode-юзер (поле `unique`).

## Регионы и компоненты

Для каждого компонента: где это веб-превью (наши токены/классы) и где это нативный Telegram (мы задаём контент, не вид).

### R1 · `tg-bar` — шапка чата (веб-превью)
| Элемент | Контент | Токены / шрифт |
|---|---|---|
| Аватар `.av` | «XC» (монограмма) | 32×32, `border-radius:50%`, фон `--xero`, текст `--cream`, `--display` 600 / 14px |
| Имя `.nm` | «XeroCode AI» | `--display` 500 / 15px, `--on-bg`, tracking −1% |
| Статус `.stat` | «● online · печатает» | `--mono` / 9.5px, цвет `--xero`, tracking +15% |
| Фон бара | — | `--bg-soft`, низ `1px --line-bg`, padding 14×18 |

> Точка `●` в статусе — символ в строке (`--mono`), НЕ lucide-иконка: это имитация нативного presence-индикатора Telegram. В реальном Telegram строку статуса («онлайн / печатает…») рисует сам клиент — мы на неё не влияем.

### R2 · `tg-msgs` — лента сообщений (веб-превью)
Контейнер: `padding:16`, `gap:12`, `min-height:360`, фон — едва заметный диагональный `linear-gradient(135deg, rgba(124,92,255,0.04), transparent)` (намёк на violet-подложку чата, не градиентная заливка площади — допустимо по бренду как 4%-вуаль).

**Пузырь `.tg-msg`** (общее): `max-width:80%`, `padding:10×14`, `border-radius:14`, 13px / LH 1.45, `--body`.
| Тип | Класс | Фон | Текст | Выравнивание | Хвостик |
|---|---|---|---|---|---|
| Бот | `.tg-msg.bot` | `--bg-soft` | `--on-bg` | `flex-start` (слева) | `border-bottom-left-radius:4px` |
| Пользователь | `.tg-msg.me` | `--xero` | `--cream` | `flex-end` (справа) | `border-bottom-right-radius:4px` |

> **Маппинг на токен-правило брифа:** пузыри `me` и кнопка отправки — единственные места violet-заливки (`--xero`) на экране, ровно по ТЗ. Бот-пузыри — нейтральный `--bg-soft`. Крупный радиус (14px) с одним «срезанным» углом (4px) = телеграм-подобный хвостик. Эмодзи в контенте бота (`💬`) — это **контент сообщения**, а не иконка UI (запрет «эмодзи как иконка» не нарушается: иконок интерфейса тут нет, кроме `→`).

В реальном Telegram пузыри рисует клиент: цвет «своих» сообщений = тема пользователя (по умолчанию сине-голубой, не violet). **Наш violet `me`-пузырь живёт только в превью** — это художественная стилизация под бренд, а не обещание, что Telegram перекрасит чат.

### R3 · `tg-input` — поле ввода (веб-превью)
| Элемент | Контент | Токены |
|---|---|---|
| Поле `.field` | плейсхолдер «Сообщение…» | `--body` 13px, текст `--on-bg-dim`, фон `--bg`, `1px --line-bg`, `border-radius:999px`, padding 8×14 |
| Кнопка `.send` | «→» | 32×32, `border-radius:50%`, фон `--xero`, символ `--cream` |
| Бар | — | фон `--bg-soft`, верх `1px --line-bg`, gap 10 |

> Стрелка `→` в превью — символ (U+2192), не lucide. В **продакшн-React** (если когда-нибудь нарисуем превью-компонент по-настоящему) заменить на `<ArrowRight/>` или `<Send/>` из `lucide-react` — по правилу проекта «все иконки из lucide-react». В реальном Telegram кнопку отправки рисует клиент.

### R4 · Кнопки-команды / быстрые ответы (**ПРЕДЛОЖЕНИЕ** — нативный Telegram, не в мокапе)
В мокапе их нет, но для удобства мобильного канала предлагаю два нативных механизма Telegram (мы задаём НАБОР и подписи, вид — клиента):

**(a) Bot Menu Button + список команд** (`setMyCommands`). При нажатии «Menu» (или вводе «/») Telegram показывает список:
```
/new      — Новый диалог
/chat     — Спросить AI
/task     — Создать задачу
/deals    — Топ сделок (CRM)
/status   — Статус команды
/link     — Привязать аккаунт XeroCode
/help     — Помощь
```
Подписи — на русском (правило «Russian UI»). Команды `/task /deals /status` уже обрабатываются в `telegram.py`.

**(b) Inline-кнопки под ответом** (`reply_markup.inline_keyboard`) — контекстные действия к конкретному сообщению. Для сценария мокапа после follow-up'а:
```
[ Серьёзнее ]  [ Короче ]  [ Сохранить в задачу ]
[ Открыть в вебе → ]
```
Нажатие → `callback_query` → бот перегенерирует/действует. «Открыть в вебе →» — `url`-кнопка (deep-link в веб-диалог). Это «быстрые ответы», заменяющие ручной ввод «сделай посерьёзнее».

**(c) Reply-keyboard** (постоянные кнопки над полем ввода) — **НЕ рекомендую** для свободного диалога: занимает место, мешает печатать. Лучше Menu-кнопка (a) + inline (b).

> Внешний вид inline/reply-кнопок полностью задаёт Telegram (тема, скругление, цвет). Наш контроль = только тексты и логика. Тексты — короткие, глагол/существительное, без хайпа.

## Поведение и интеракции

### Команды бота (реальные + предлагаемые)
| Команда | Поведение | Статус в коде |
|---|---|---|
| `/start` | Онбординг: если есть `link_<token>` → привязать; иначе — приветствие + предложение привязать | заглушка (привязка не дописана) |
| `/new` | Сбросить контекст диалога, начать новый | **ПРЕДЛОЖЕНИЕ** (нет в коде; нужно хранилище сессии) |
| `/chat <текст>` | Явный AI-запрос | через free-text fallback уже работает |
| `/task <текст>` | Создать задачу в XeroCode (`Task`, status `backlog`) | **РЕАЛИЗОВАНО** (`telegram.py:127`) |
| `/deals` | Топ-5 сделок CRM по сумме, по `organization_id` | **РЕАЛИЗОВАНО** (`:146`) |
| `/status` | Статистика орг: команда / задачи / сделки | **РЕАЛИЗОВАНО** (`:166`) |
| `/link` | Запустить привязку аккаунта | **ПРЕДЛОЖЕНИЕ** |
| `/help` | Список возможностей | **ПРЕДЛОЖЕНИЕ** |
| свободный текст | → AI (`_call_ai`, `prefer_premium=False`), ответ обрезается до 4000 символов | **РЕАЛИЗОВАНО** (`:199`) |

### Свободный диалог (ядро сценария мокапа)
- Пользователь пишет задачу естественным языком («Напиши follow-up…») → бот зовёт `_call_ai` (тот же бэкенд, что и веб-чат A03) → отвечает текстом.
- **Итерация тона** («Сделай посерьёзнее») → бот учитывает предыдущий ответ как контекст и переписывает. ⚠️ Сейчас `_call_ai` вызывается **без истории** (один system + один user) — итерация «посерьёзнее» в текущем коде не сохранит предыдущий follow-up. Чтобы воспроизвести сценарий мокапа, нужна **память диалога** (см. «Данные» / «Открытые вопросы»).
- **Split-ответ.** В мокапе бот отвечает двумя пузырями подряд («Понял, более деловой тон:» + сам текст). Воспроизводится двумя `sendMessage` подряд — приём «человечности» (бот «думает вслух», потом выдаёт результат). **ПРЕДЛОЖЕНИЕ** закрепить как паттерн: короткая реплика-подтверждение + результат отдельным сообщением, чтобы длинный текст было удобно копировать целиком.

### Выбор воркспейса в боте (**ПРЕДЛОЖЕНИЕ**)
Бот = в первую очередь **Чат (01)** + командные срезы CORP (задачи/CRM/статус). Полноценного «переключателя 8 воркспейсов» в боте быть не должно (UX мессенджера это не держит). Вместо этого:
- **AI-роутер** (как в A12 Empty state «AI подберёт воркспейс»): по тексту бот сам решает — это AI-ответ, задача (`/task`), картинка или «это лучше в вебе → вот ссылка».
- Если запрос требует веб-воркспейса (DAG, batch-картинки, документ-превью) → бот отвечает текстом + inline-кнопкой `Открыть в XeroCode →` (deep-link на нужный экран).

### Прикрепление файла / голоса (**ПРЕДЛОЖЕНИЕ** — нативный Telegram)
- **📎 Файл/фото** (нативная скрепка Telegram): пользователь шлёт фото/документ → бот принимает `message.photo`/`message.document`, может: распознать (картинка → описание/OCR), принять документ в Базу знаний (A07), или как вложение к задаче. Сейчас webhook парсит только `message.text` — медиа не обрабатывается (расширение бэкенда).
- **🎤 Голос** (нативная запись Telegram): `message.voice` → транскрипция (наш voice-роут уже есть в проекте: `routes/voice.py`) → дальше как текст. Состояние «расшифровываю…» (см. «Состояния»).

### Inline-режим бота (**ПРЕДЛОЖЕНИЕ**, низкий приоритет)
`@xerocode_bot <запрос>` прямо в ЛЮБОМ чате Telegram → бот предлагает варианты ответа (например, готовый follow-up) для вставки в текущую переписку. Мощно для «написать клиенту, не покидая чат с клиентом», но требует отдельной обработки `inline_query` и осторожности с приватностью (см. Edge cases). Помечено как поздняя фича.

### Привязка через токен/QR (механика)
1. Веб (A09) генерит `link_<token>` (TTL ~5 мин, как device-code).
2. Юзер → `t.me/xerocode_bot?start=link_<token>` (по ссылке или QR).
3. Бот `/start` парсит токен, валидирует, пишет `TelegramLink(user_id, telegram_user_id, telegram_username)`.
4. Бот: «✓ Готово, [Имя]. Аккаунт XeroCode привязан. Напишите задачу или /help.»
5. Веб (если открыт) поллит статус и показывает «✓ Telegram привязан · @username».

| **Онбординг / не привязан** | `/start` → приветствие + кнопка/инструкция привязки. Команды по данным (`/deals`, `/status`) отвечают «Аккаунт не привязан. /link для привязки.» | `/deals`/`/status` уже возвращают «not linked»; приветствие — заглушка |
| **Привязан, idle** | Приветствие «Что генерируем сегодня?» + Menu-кнопка с командами | мокап-строка `tg-msg.bot` |
| **Печатает / генерация** | Нативный индикатор Telegram **«печатает…»** (`sendChatAction: typing`) пока модель думает. В превью — статус `● online · печатает` в `tg-bar` | `sendChatAction` — **ПРЕДЛОЖЕНИЕ** добавить в webhook |
| **Голосовой ввод → транскрипция** | После `voice` — нативный action «расшифровываю…» (`upload_voice`/`typing`), затем эхо-текст «Вы сказали: …» (опц.) + ответ | **ПРЕДЛОЖЕНИЕ** |
| **Длинный ответ** | Разбить на части ≤4096 символов (Telegram-лимит) ИЛИ обрезать с «… (полностью — в вебе →)». Сейчас код **жёстко режет на 4000** | `reply = ai_response[:4000]` (`:209`) |
| **Ошибка модели / провайдера** | «Не получилось сгенерировать ответ, попробуйте ещё раз 🙏» (текст, без стек-трейса). Сейчас ошибка только логируется (`print`), пользователю ничего не уходит | `except → print` (`:212`) — **нужен ответ юзеру** |
| **Лимит (тариф/баланс)** | «Достигнут лимит на вашем тарифе. BYOK снимает лимит — подключите ключ в Настройках → API.» + inline-кнопка `Открыть Настройки →` | **ПРЕДЛОЖЕНИЕ** (увязать с биллингом, как A09 «BYOK = ∞») |
| **Бот остановлен/заблокирован юзером** | Telegram вернёт ошибку доставки — гасим тихо, чистим сессию | нативное поведение |

> Все пользовательские тексты — **на русском** (правило проекта). Сейчас часть командных ответов в коде на английском («Top deals», «No deals found», «Account not linked», «XeroCode Status») — это **баг локализации**, в дизайне фиксируем русский.

## Граница бот vs веб

| Возможность | В боте | Только в вебе |
|---|---|---|
| **Чат с моделью (01)** | ✅ полностью (free-text → `_call_ai`) | расширенно: выбор из 14 моделей, ветки/branching, citations, token-счётчик (A03) |
| **Текст-генерация (02)** | ✅ короткие тексты в переписке (follow-up, посты) | длинные документы, редактор, AI-на-лету |
| **Картинки (03)** | ⚠️ **ПРЕДЛОЖЕНИЕ** — простая генерация по тексту, бот шлёт картинку файлом | стили/эталоны/размеры/batch, грид, лимит-счётчик (A04) |
| **Задачи / CORP** | ✅ `/task`, `/deals`, `/status` (срезы данных) | полноценные CRM/Kanban/документооборот/HR (A07-корп) |
| **База знаний (07)** | ⚠️ **ПРЕДЛОЖЕНИЕ** — приём файла в KB; RAG-запрос текстом | загрузка, коннекторы (1С/Bitrix/Notion…), карточки источников |
| **Код (05)** | ❌ (фрагменты текстом — можно, но не редактор) | редактор, diff, Apply changes, табы файлов |
| **Видео (06) / Звук** | ❌ | Sora/Runway/Veo, Suno/ElevenLabs |
| **Документы-превью (06-docs)** | ❌ (только ссылка) | Cream-экран, рендер PPTX/DOCX/XLSX |
| **Оркестрация / DAG (08)** | ❌ (только триггер/нотификации — **ПРЕДЛОЖЕНИЕ**) | визуальный no-code билдер (A08) |
| **Настройки / BYOK (A09)** | ❌ (только deep-link «Открыть Настройки →») | вся настройка ключей, тарифа |
| **Привязка аккаунта** | ✅ приём deep-link/кода | генерация токена/QR (A09) |

**Принцип границы:** бот делает то, что **умещается в реплику** и ценно «на ходу». Всё, что требует холста, грида, многошагового UI или тонкой настройки — кнопка `Открыть в XeroCode →`. Бот может **уведомлять** о событиях из веба (готов DAG-дайджест, новая сделка) — это его сильная сторона как push-канала (**ПРЕДЛОЖЕНИЕ**, увязать с `routes/push.py`).

## Адаптив

Экран изначально **mobile-first** — это и есть мобильный канал. Адаптив тут не «брейкпоинты нашей сетки», а **различия клиентов Telegram**:

- **Веб-превью (`tg-frame` 320px)** — фиксированная ширина «телефона» внутри `browser-frame`. На узких вьюпортах лендинга `browser-frame` сжимается, телефон остаётся 320px по центру (`margin:0 auto`). На очень узком (<360px) — допустимо отдать `tg-frame` `width:100%; max-width:320px`. **ПРЕДЛОЖЕНИЕ** для мобильной версии лендинга.
- **iOS vs Android (реальный бот).** Мы не управляем этим — но учитываем:
  - **iOS**: пузыри/скругления/шрифт по гайдам Apple, «своя» сторона справа, цвет — тема пользователя; reply-keyboard выезжает иначе; safe-area снизу.
  - **Android**: Material-пузыри, FAB-подобная отправка, длинный тап = контекстное меню.
  - **Desktop/Web Telegram**: горячие клавиши, drag-n-drop файла, inline-кнопки шире.
  Вывод для дизайна: **не зашиваем верстку под конкретный клиент** — формируем семантику (текст + кнопки + медиа), вид отдаёт Telegram. Тестировать на iOS, Android, Desktop, Web (4 клиента).
- **Темы пользователя.** Тёмная/светлая тема Telegram переопределит фоны — поэтому в реальном боте **не полагаемся на цвет пузыря** для смысла (например, не «красный = ошибка»): смысл несёт текст/эмодзи в контенте.

|---|---|---|

| **Привязка-токен (link/QR)** | ❌ **нет** | по образцу `cli_auth._pending` (in-memory или Redis), TTL ~5 мин. **ПРЕДЛОЖЕНИЕ** |
| **История бота** | частично — задачи/сделки пишутся в общие таблицы (`tasks`, видна в вебе) | переписка-история не хранится (stateless webhook) |

**Идентичность.** `chat_id` (куда отвечать) ≠ `tg_user_id` (кого матчим с `TelegramLink`). В личке совпадают, в группе — нет (см. Edge cases).

## Edge cases

- **Не привязан + командный запрос** (`/deals`, `/status`) → вежливый отказ «Аккаунт не привязан. /link» (уже есть, но по-английски — локализовать).
- **Длинный ответ >4096** → разбивка на части или обрезка с «полностью — в вебе →». Сейчас режется на 4000 без уведомления → **дизайн-требование**: добавлять хвост «…(сообщение сокращено)».
- **Медиа без обработчика** (фото/документ/стикер/гео) → сейчас игнорируется (`text` пустой → `return ok`). **Дизайн-требование**: явный ответ «Пока умею текст и голос. Фото/файлы — скоро 🙏» вместо тишины. Полноценный приём медиа — **ПРЕДЛОЖЕНИЕ**.
- **Голос** → транскрипция (`routes/voice.py`); если не распозналось — «Не расслышал, повторите текстом?».
- **Групповой чат** (**ПРЕДЛОЖЕНИЕ / осторожно**): бот в группе видит сообщения по privacy-режиму. Решения: (1) отвечать **только на @упоминание/команду**, (2) `tg_user_id ≠ chat_id` — команды по данным должны матчить **отправителя**, не группу; неуместно сыпать чужой CRM в общий чат → CORP-команды в группах **запретить** или отвечать в ЛС. Привязка в группе невозможна (нужна личка).
- **Несколько Telegram-аккаунтов на один XeroCode** → `telegram_user_id unique` запрещает дубль связи; повторный `/start link_` с другого TG — «Этот аккаунт уже привязан к другому Telegram».
- **Просроченный/чужой link-токен** → «Ссылка устарела. Сгенерируйте новую в Настройках XeroCode.»
- **Бот заблокирован/остановлен юзером** → доставка падает; гасим, при следующем `/start` — заново.
- **Rate-limit Telegram** (≈30 msg/sec на бота, ~1 msg/sec в чат) → очередь/бэкофф при массовых уведомлениях.
- **Ошибка AI-провайдера** → пользователю дружелюбный текст (сейчас — тишина + `print`; **баг**, чинить).
- **Webhook secret не совпал** → отбой (`{"ok": false}`); реализовано через `X-Telegram-Bot-Api-Secret-Token`.
- **Дубли апдейтов** (Telegram ретраит при не-200) → идемпотентность по `update_id` (**ПРЕДЛОЖЕНИЕ**, чтобы не плодить задачи на ретрае).

## Открытые вопросы / ПРЕДЛОЖЕНИЯ

1. **Объём функций бота.** Подтвердить границу: «Чат (01) + командные срезы CORP (задачи/CRM/статус) + уведомления», остальное — deep-link в веб. Картинки/KB в боте — да/нет на первый релиз? *(ПРЕДЛОЖЕНИЕ: чат+команды на MVP, картинки/KB — фаза 2.)*
2. **Привязка — основной способ.** Deep-link `?start=link_<token>` + QR из A09 (ПРЕДЛОЖЕНИЕ как основной) против только Login Widget. Нужен endpoint генерации токена (нет). Где в A09 живёт кнопка «Привязать Telegram» + QR?
3. **Память диалога.** Без неё нет итерации тона/`/new` (ключ сценария мокапа). Хранилище: in-mem (просто, теряется при рестарте) vs Redis (надёжно). TTL диалога? *(ПРЕДЛОЖЕНИЕ: Redis, TTL 24ч.)*
4. **Голос/транскрипция.** Включаем `voice` → `routes/voice.py` на MVP? Эхо-текст распознанного показывать? *(ПРЕДЛОЖЕНИЕ: да к голосу — это сильный мобильный жест; эхо опционально.)*
5. **Inline-режим** (`@xerocode_bot` в чужих чатах). Мощно для follow-up «не выходя из чата с клиентом», но риск приватности (CORP-данные в чужом чате — запретить). *(ПРЕДЛОЖЕНИЕ: только генеративные действия, без данных орг; поздняя фаза.)*
6. **Групповые чаты.** Поддерживаем? Если да — только @упоминание/команды, CORP-данные только в ЛС отправителя. *(ПРЕДЛОЖЕНИЕ: личка — основной режим; группы — позже, ограниченно.)*
7. **Push из веба в бот.** Бот как канал нотификаций (готов дайджест DAG, новая сделка, упоминание). Связать с `push.py`. *(ПРЕДЛОЖЕНИЕ: да — это уникальная ценность мессенджер-канала.)*
8. **Лимиты/тариф в боте.** Как показываем исчерпание лимита и BYOK=∞ (тезис A09)? inline-кнопка в Настройки. *(ПРЕДЛОЖЕНИЕ.)*
9. **Локализация (баг, не вопрос).** Все ответы бота → русский; сейчас часть на английском в `telegram.py`.
10. **Один бот на всех vs бот-на-организацию.** В коде сосуществуют: глобальный `@xerocode_bot` (`settings.telegram_bot_token`, Login Widget) и пер-орг боты (`Integration.config.bot_token`). 11. **Идемпотентность апдейтов** по `update_id` (чтобы ретраи Telegram не плодили задачи). *(ПРЕДЛОЖЕНИЕ.)*
