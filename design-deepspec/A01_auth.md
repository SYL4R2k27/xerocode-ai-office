# A01 · Авторизация — `/auth`

> Глубокая дизайн-спецификация одного экрана XeroCode v3.
> Источник истины: `XEROCODE_FULL.html` строки 1840–1886 + `xerocode-v3.css` строки 650–761 (блок `.auth-*`).
> Все цвета — только именами CSS-переменных. Все иконки — `lucide-react`. Весь UI-текст — по-русски.
> Запрещёнка соблюдена: без курсива/serif, без больших градиентов, без glass/glow/частиц, без киберпанка и хайпа, без спиннеров дольше 200ms.

---

## Назначение

Точка входа в продукт. Единственный публичный экран приложения (наравне с лендингом), задача которого — за один-два ввода пустить существующего пользователя внутрь и не отпугнуть нового. Экран должен мгновенно сообщить «это серьёзный российский AI-офис, а не игрушка»: спокойный карбон, точная типографика Space Grotesk, trust-маркеры `SOC 2 · ФЗ-152`, минимум движения. Левая половина — это бренд-витрина (продаёт «один продукт — восемь воркспейсов»), правая — рабочая форма входа. Цель — нулевое трение для повторного входа (email + пароль + Enter) и понятный путь к регистрации и OAuth.

## Доступ

**public** — экран доступен без токена. Если в `localStorage` уже лежит валидный `ai_office_token`, пользователь не должен видеть `/auth` — его редиректит в `/dashboard` (см. «Поведение → Авторедирект»). Сюда же приземляются три служебных сценария: обычный вход, возврат после OAuth (`/auth/callback#token=…`) и fall-through из CLI-логина (`/cli-login`, `/cli/approve`) — после успешного входа управление возвращается инициатору.

## Раскладка

Корневой контейнер — **`.auth-frame`**: CSS Grid, две равные колонки `1fr 1fr`, `min-height: 600px`. На desktop это полноэкранный split 50/50, обёрнутый в продуктовый каркас (в мокапе — `browser-frame`; в реальном приложении это просто полноэкранная страница, топ-бар не нужен — экран public).

- **Левый регион `.auth-left`** — бренд-панель. Поверхность `--bg` (Carbon, самый тёмный). Внутренние поля `60px 56px`. Flex-колонка с `justify-content: space-between` → три яруса прижаты: бренд сверху, слоган по центру массы, trust-meta снизу.
- **Правый регион `.auth-right`** — форма. Поверхность `--bg-soft` (на полтона светлее левой — единственная граница между половинами создаётся контрастом поверхностей, без линии-разделителя). Поля `60px 56px`. Flex-колонка с `justify-content: center` → форма вертикально центрирована.
- **Форма `.auth-form`** — `max-width: 360px`, `width: 100%`, `margin: 0 auto` (центрируется в правой половине). Все поля и кнопки внутри — на всю ширину 360px.

Граница между половинами — стык двух поверхностей (`--bg` слева ↔ `--bg-soft` справа). Никаких вертикальных хайрлайнов и теней между колонками.

## Поведение и интеракции

### Авторедирект (до рендера)
- При маунте читаем `ai_office_token` из `localStorage`. Если токен есть и не истёк → `location.replace('/dashboard')`, форму не показываем (нет «вспышки» логина).
- Если URL = `/auth/callback` ИЛИ `hash` начинается с `#token=` → извлекаем JWT из хэша, кладём в `ai_office_token`, чистим хэш, редиректим в `/dashboard`. Это возврат из OAuth.

### Поле email (`.auth-input`)
- **hover:** курсор text; рамка без изменения.
- **focus:** рамка → `--xero` (1px, замена `--line-bg-strong`). Допустимо мягкое кольцо `box-shadow: 0 0 0 3px var(--xero-mute)` — это `rgba(124,92,255,.15)`, плоское, НЕ glow (ПРЕДЛОЖЕНИЕ: явно зафиксировать как стандарт focus-ring).
- **keyboard:** `Enter` в любом поле сабмитит форму. `Tab` → password. Поле первым получает автофокус при открытии (если не пришли из OAuth).
- **валидация:** при сабмите — не пустое; формат email (наличие `@` и домена). Пустое или кривое → инлайн-ошибка под полем / в общем контейнере, рамка поля → `--sylar` (Sienna как «ошибочный» акцент, НЕ красный из-за бренда). Валидация по `blur` — мягкая (не блокирует), при сабмите — жёсткая.

### Поле password (`.auth-input type=password`)
- **focus/keyboard:** как email.
- **toggle видимости** (ПРЕДЛОЖЕНИЕ #1): иконка `Eye`/`EyeOff` (lucide, 16px) в правом краю поля, `--on-bg-dim` → на hover `--on-bg-mute`. Клик переключает `type` password↔text. Доступно с клавиатуры (`button`, `aria-label="Показать пароль"`).
- **валидация:** не пустое; минимум 6 символов (правило бэка). Меньше 6 → инлайн-ошибка «Пароль должен быть не менее 6 символов».

### Кнопка «Войти →» (`.auth-btn`)
- **hover:** яркость заливки +6–8% (`filter: brightness(1.06)`) ИЛИ затемнение `--xero` на ~6%; переход 180ms `--ease`. Никакого scale-bounce внутри продукта (magnetic — приём только для лендинговых pill-CTA).
- **active/press:** `transform: translateY(1px)` или brightness −4%.
- **focus-visible:** кольцо `0 0 0 3px var(--xero-mute)`.
- **click:** валидирует поля → если ок, переводит форму в `loading`, дизейблит кнопку, шлёт `POST /api/auth/login {email,password}`.
- **keyboard:** активируется `Enter`/`Space`.

### Разделитель «или» (`.auth-divider`)
- Декоративный, не интерактивный. Два хайрлайна `--line-bg` + центрированный mono-текст.

### OAuth-кнопки (`.auth-oauth button`)
- **hover:** рамка `--line-bg-strong` → `--xero` (или `--line-bg-strong` усиливается); фон `--bg` → лёгкая подсветка `--xero-mute`; переход 200ms.
- **click Google:** `location.href = '/api/auth/oauth/google'` → бэк делает 302 на провайдера → callback вернёт нас на `/auth/callback#token=…`.
- **click Telegram:** открывает Telegram Login popup (виджет `telegram-widget.js`, `Telegram.Login.auth({bot_id,…})`); при успехе POST на `/api/auth/oauth/telegram/verify`, токен → `ai_office_token` → редирект. Пока скрипт грузится — на клик показать тихий тост «Telegram ещё грузится».
- **keyboard:** обе фокусируемы, `Enter`/`Space`.

### Ссылка «Регистрация» (`.auth-bottom a`)
- **hover:** `--xero` → `--xero-soft`; подчёркивание появляется на hover (по умолчанию его нет).
- **click:** переключает форму в режим «Регистрация» (in-place, без перехода на другой URL) ИЛИ ведёт на `/auth?mode=register`. In-place предпочтительнее (текущий прод так и делает: `mode: login|register`). В режиме регистрации:
  - `.lab` → «▸ РЕГИСТРАЦИЯ», `h2` → «Создать<br>аккаунт.»;
  - появляются поля «Имя» и «Инвайт-код (бета-тест)» + индикатор силы пароля;
  - кнопка → «Создать аккаунт →»; `.auth-bottom` → «Уже есть аккаунт? <a>Войти</a>».
- Переключение режима — `AnimatePresence` кросс-фейд + лёгкий сдвиг по X (≤20px, ≤250ms, `--ease`).

### Глобальная клавиатура
- `Enter` где угодно в форме → сабмит активного режима.
- `Esc` — без действия (некуда закрывать; экран — корень).
- Порядок `Tab`: email → password → (toggle-пароля) → «Забыли пароль?» → «Войти» → Google → Telegram → «Регистрация».

## Адаптив

**Desktop ≥1280** — `.auth-frame` = два равных столбца `1fr 1fr`, `min-height:600px` → в реальном приложении растягивается на `100vh`. Левая бренд-панель и правая форма видны одновременно. Поля 60/56px.

**Tablet 768–1279** — split сохраняется (`1fr 1fr`), но поля сжимаются: `--pad-x` уводит к нижней границе `clamp(20px,4vw,64px)` ≈ 40–48px. Слоган `.auth-quote` остаётся 30px либо мягко уменьшается до ~26px. Форма по-прежнему `max-width:360px`, центрирована. На узкой границе диапазона (≈768–860px) допустимо начать прятать `.auth-meta` или сжимать слоган в 2 строки.

**Mobile <768** (брейкпоинт CSS `max-width:760px`) — `.auth-frame` → `grid-template-columns: 1fr` (одна колонка), `.auth-left` и `.auth-right` → `padding: 40px 28px`. Раскладка:
- Левая бренд-панель сворачивается в компактную шапку: остаётся `.auth-brand` (XeroCode•) сверху; `.auth-quote` сокращается до 1–2 строк или прячется; `.auth-meta` уходит в подвал под формой (ПРЕДЛОЖЕНИЕ #7). Поверхность шапки `--bg`, форма ниже на `--bg-soft` — вертикальный стек.
- Форма растягивается на ширину экрана минус поля (28px по бокам), `max-width:360px` всё ещё ограничивает на крупных телефонах.
- OAuth-кнопки остаются в 2 колонки (помещаются); при очень узких экранах (<360px) допустим стек в 1 колонку.
- Клавиатура телефона не должна перекрывать кнопку «Войти» — форма центрируется выше середины при фокусе (учитывать `visualViewport`).

## Данные

**Поля формы (вход):**
- `email` — string, `type=email`, обяз., autocomplete `email`.
- `password` — string, `type=password`, обяз., min 6, autocomplete `current-password`.

**Поля формы (регистрация — смежный режим, не в мокапе входа):**
- `name` — string, обяз., autocomplete `name`.
- `email`, `password` (autocomplete `new-password`).
- `inviteCode` — string, обяз. (бета-гейт; бэк сверяет с `INVITE_CODE` из настроек; несовпадение → 403/ошибка «Введите инвайт-код…»).

**OAuth-провайдеры (в мокапе ровно два):**
- **Google** → `GET /api/auth/oauth/google` → 302 → callback → `/auth/callback#token=`.
- **Telegram** → Telegram Login Widget popup → `POST /api/auth/oauth/telegram/verify`.
- *(В текущем проде живут ещё GitHub и Yandex — в v3-мокапе их НЕТ. Решение: оставить только Google+Telegram по мокапу, либо вернуть 4 — см. ПРЕДЛОЖЕНИЕ #5.)*

**Trust-маркеры (`.auth-meta`):** `V 3.0` (версия), `SOC 2` (аудит безопасности), `ФЗ-152` (152-ФЗ «О персональных данных» — РФ-комплаенс, ключевой сигнал для российской аудитории).

**Бэкенд-контракты (для маппинга):**
- `POST /api/auth/login` → `TokenResponse { access_token, refresh_token }`.
- `POST /api/auth/register` (с `invite_code`) → тот же `TokenResponse`.
- `POST /api/auth/refresh` → новый `TokenResponse`.
- Токен сохраняется в `localStorage` под ключом **`ai_office_token`** (единый ключ по всему проекту — правило проекта).

## Edge cases

- **Неверный пароль** — бэк 401 → контейнер-ошибка «Неверный email или пароль» (не раскрывать, что именно неверно — антиперебор). Поля не очищаются, фокус возвращается в password.
- **Заблокирован аккаунт** — бэк 403 → «Аккаунт заблокирован. Напишите в поддержку» + (опц.) ссылка на поддержку. OAuth-кнопки остаются активны.
- **Rate-limit / слишком много попыток** — бэк 429 → «Слишком много попыток. Повторите через минуту»; кнопку «Войти» можно временно задизейблить с обратным отсчётом (ПРЕДЛОЖЕНИЕ).
- **2FA** — в мокапе НЕТ. **ПРЕДЛОЖЕНИЕ #9:** для Enterprise/команд до 50 seats предусмотреть шаг 2FA как отдельное состояние формы (после успешного пароля бэк отвечает `mfa_required` → форма заменяет поля на одно поле «Код из приложения», 6 цифр, `tabular-nums`, иконка `ShieldCheck`; «Войти →» → «Подтвердить →»). Сейчас — не реализуем, только закладываем место в потоке.
- **Длинный email** — поле `overflow` скрывает, текст скроллится внутри; `tabular-nums` держит цифры ровно; перенос не делаем (однострочный input). Плейсхолдер-домен `@company.ru` намекает на корпоративный кейс.
- **Медленная сеть** — после сабмита кнопка в `loading` (disabled, лейбл «Вход…»). Если ответ дольше ~8с — показать тихую подсказку «Проверяем… соединение медленное» в контейнере (нейтральный тон, НЕ ошибка). Никаких бесконечных крутящихся спиннеров (бренд: спиннеры ≤200ms).
- **OAuth отменён/закрыт popup** — тихий тост «Авторизация отменена», форма остаётся как была.
- **Невалидный/просроченный токен в `#token=`** — если callback принёс битый JWT, не редиректим в dashboard, а показываем `/auth` с ошибкой «Сессия не удалась, войдите снова».
- **Автозаполнение браузера** — стили input не должны ломаться от `:-webkit-autofill` (переопределить фон автозаполнения на `--bg`, текст `--on-bg`).
- **Двойной сабмит** — `loading`-гард: повторный `Enter`/клик во время запроса игнорируется.

## Открытые вопросы / ПРЕДЛОЖЕНИЯ

1. **ПРЕДЛОЖЕНИЕ — toggle видимости пароля.** В мокапе поля нет, но это базовый UX. Добавить `Eye/EyeOff` (lucide, 16px, `--on-bg-dim`) в правом краю password-input. *Вне мокапа.*
2. **ПРЕДЛОЖЕНИЕ — лидирующие иконки полей и стрелка кнопки.** `Mail` в email-поле и `ArrowRight` в кнопке вместо текстовой `→` — для консистентности с lucide. Альтернатива (ближе к мокапу): оставить `→` текстом, иконки полей не добавлять. **Вопрос: иконизировать поля или держать чистый мокап?**
3. **Вопрос — формат trust-meta.** В мокапе `V 3.0 · SOC 2 · ФЗ-152` без mono-маркера `▸`. Добавлять ли префикс `▸` (как у других mono-лейблов) или оставить «голую» строку? Также: **верны ли заявки SOC 2 / ФЗ-152 юридически на текущем этапе** — если нет, не вводить пользователя в заблуждение (заменить на честные «Шифрование · РФ-хостинг» до получения сертификатов).
4. **ПРЕДЛОЖЕНИЕ — «Забыли пароль?».** В мокапе нет. Бэк восстановления пароля, судя по проду, ещё не готов («Скоро!»-тост). Решить: (а) скрыть ссылку до готовности эндпоинта, или (б) показать ссылку → тост-заглушка. Рекомендация: скрыть, пока нет backend-flow, чтобы не плодить мёртвые ссылки.
5. **Вопрос — состав OAuth: 2 или 4 провайдера?** Мокап показывает ровно **Google + Telegram**. Прод-код имеет **Google + GitHub + Yandex + Telegram**. Для российской аудитории Yandex логичен; GitHub — нишево (нужен `/code`-аудитории). **Решение нужно от владельца:** строго по мокапу (2) ИЛИ вернуть 4 в grid (тогда `.auth-oauth` → `repeat(2,1fr)` в 2 ряда, либо иконочные кнопки в один ряд). Если иконки фирменных брендов — это исключение из «только lucide» (бренд-логотипы провайдеров обычно держат как SVG; зафиксировать как разрешённое исключение).
6. **ПРЕДЛОЖЕНИЕ — режим «Регистрация» (поля Имя + Инвайт-код + сила пароля).** Не в мокапе входа, но есть в проде и нужен функционально (бета-гейт). Шкала силы пароля сейчас использует `--accent-rose/amber/blue/teal` (V2) → перевести на v3: Слабый `--sylar`, Средний `--ws-video` (amber), Хороший `--ws-text` (cyan), Отличный `--ws-code` (green). **Вопрос: спецификация регистрации — отдельным экраном A01b или как состояние A01?** (Рекомендация: одно состояние, in-place переключение.)
7. **ПРЕДЛОЖЕНИЕ — мобильная бренд-панель.** На <768 левая панель сворачивается. Что оставить: только `XeroCode•` сверху + trust-meta в подвал? Или мини-слоган в 1 строку? Нужен явный мобильный макет левой половины (в мокапе только desktop split).
8. **ПРЕДЛОЖЕНИЕ — убрать V2-моушн, противоречащий бренду.** Текущий прод: бесконечный `animate-spin` (Loader2), shake ошибки, floating blobs, gradient text. Всё это — прямые нарушения запрещёнки v3 (спиннеры >200ms, glow/particles, большие градиенты). Заменить: loading → смена лейбла «Вход…» / пульс-точки; ошибка → мягкий fade-in без тряски.
9. **ПРЕДЛОЖЕНИЕ — 2FA как будущее состояние.** Для команд до 50 seats и Enterprise предусмотреть TOTP-шаг: бэк возвращает `mfa_required`, форма показывает одно поле «Код из приложения» (6 цифр, `tabular-nums`, иконка `ShieldCheck`, кнопка «Подтвердить →»). Сейчас не строим — резервируем место в потоке аутентификации, чтобы редизайн не пришлось ломать позже.
10. **Вопрос — авторедирект и «вспышка» формы.** Подтвердить: проверка `ai_office_token` и редирект в `/dashboard` происходят **до** первого пейнта формы (иначе мелькает логин у уже залогиненного). Это поведенческое требование, не визуальное — зафиксировать в реализации.
