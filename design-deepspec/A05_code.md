## Назначение

Рабочее пространство для кода: пользователь видит файл проекта, AI читает контекст и **предлагает diff**, человек жмёт **Apply** — изменение применяется. Это не полноценная IDE-замена, а «AI-парный программист»: ядро воркспейса — паттерн **«AI предлагает diff → человек принимает / отклоняет»**. Поддержка multi-language (в мокапе Python), CLI с прямой авторизацией, PR/MR-helper и code review (см. §3 ИА брифа, строка 96).

Три канала-входа в один и тот же воркспейс (тег экрана `/code · CLI · MCP · IDE`, строка 2182):
- **CLI** — `xerocode-cli` (отдельный продукт-терминал, см. коммиты `caba851`, `d33a15b`): работа из консоли пользователя, прямая авторизация по токену.
- **MCP** — XeroCode выступает MCP-сервером/клиентом: внешний агент (или IDE) дёргает инструменты воркспейса по протоколу.
- **IDE** — расширение/мост к редактору пользователя (VS Code-класс), где diff применяется уже на реальном файле на диске.

Веб-экран A05 (эта спека) — «головной» интерфейс воркспейса: просмотр файла, табы, запуск, diff, лента AI-предложений. Тяжёлое редактирование живёт в IDE пользователя, веб — **read-focused просмотр + приём предложений** (особенно на мобиле).

## Доступ

- **Тарифы:** доступен соло-пользователям и SMB (до 50 seats); BYOK на всех тарифах (`--xero` тезис «BYOK = ∞ запросов»). В отличие от воркспейса 07 (Корпорат., только Enterprise) — Код доступен без Enterprise-гейта.
- **Org-isolation:** все запросы файлов/прогонов/предложений идут с `organization_id`; чужой файл по прямому ID не отдаётся (жёсткое правило бэка). Сиденья команды видят общие файлы проекта согласно ролям.
- **CLI/MCP/IDE-авторизация:** канал авторизуется тем же `ai_office_token` (единый ключ localStorage в вебе → bearer-токен / device-token для CLI). Прямая авторизация CLI = одноразовый device-flow или вставка токена; токен скоупится на организацию и воркспейс.
- **Навигация:** пункт «Код» в `app-side` → секция «Воркспейсы», состояние `.active` (мокап строка 2198). Активный пункт: фон `--xero-mute`, текст `--xero`, иконка `--xero` (CSS 564–568).

## Поведение и интеракции

- **Переключение табов:** клик по `code-tab` → меняется содержимое `code-body`, заголовок toolbar (`h2`) синхронизируется с активным файлом, обновляется нумерация строк. Анимация перехода — без сложного motion; допускается мгновенная смена либо короткий fade `--ease` ≤180ms. Незакоммиченный diff при переключении табов **сохраняется** в состоянии (не теряется), индикатор `●` остаётся на «грязном» табе.
- **Run → вывод:**
  - Клик `▸ Run` запускает активный файл/проект через бэкенд-исполнитель (sandbox).
  - Кнопка переходит в loading-состояние: спиннер ≤200ms (правило «спиннеры не дольше 200ms» — далее это уже прогресс-стрим, а не спиннер). Иконка `Play` → `Loader2` (анимация spin) или `Square` (Stop, чтобы прервать).
  - **run-output (ПРЕДЛОЖЕНИЕ — региона нет в мокапе):** панель вывода под code-frame (или вместо AI-suggestion, табами «Suggestion / Output / Problems»). Моно-консоль: stdout `--on-bg`, stderr/ошибки `--ws-images` (pink) или красный из палитры; exit-code бейдж (0 = `--ws-code` green, ≠0 = pink). Стрим строк по мере выполнения.
- **Diff → режим сравнения:**
  - Клик `Diff` переключает `code-body` в **diff-view**: либо unified (один столбец с маркерами), либо side-by-side (два столбца «было/стало»).
  - **Цвета diff:** added — фон `rgba(16,185,129,.12)` на базе `--ws-code` + знак `+` (`--ws-code`); removed — фон на базе pink/red + знак `−`; контекст — обычный `--on-bg`. Левый жёлоб (gutter) с маркерами `+`/`−`, mono.
  - В diff-режиме toolbar-кнопка `Diff` подсвечена (active). Повторный клик / `Esc` → выход в обычный просмотр.
  - **ПРЕДЛОЖЕНИЕ:** переключатель «unified / split» (`lucide-react` `Rows`/`Columns`), навигация по чанкам `ChevronUp`/`ChevronDown` «следующее изменение».
- **Apply → коммит изменений:**
  - `+ Apply changes` применяет текущий предложенный diff к файлу (в вебе — к рабочей копии воркспейса; через IDE-мост — к файлу на диске пользователя).
  - После Apply: AI-suggestion-карточка сворачивается/сменяется тостом «Изменения применены» (`lucide-react` `Check`, `--ws-code`); индикатор `●` грязного файла гаснет; diff-режим закрывается; (ПРЕДЛОЖЕНИЕ) появляется «Отменить» (`Undo2`) на короткое время.
  - **ПРЕДЛОЖЕНИЕ — git-связь:** Apply пишет в рабочую копию, отдельная кнопка `Commit`/`Создать PR` (`GitPullRequest`) — для PR/MR-helper из спеки воркспейса. Граница: Apply ≠ git-commit; Apply = «принять правку в файл».
- **Принять / отклонить suggestion:**
  - «Применить» (в карточке) = scoped Apply этого предложения.
  - «Отклонить» = карточка уходит (slide/fade ≤180ms `--ease`), предложение помечается dismissed (не всплывает повторно для той же ревизии).
  - «Показать diff» = открывает diff-view именно по этому предложению (мост карточка↔diff).
- **Связь CLI / MCP / IDE — авторизация и синхронизация:**
  - **Авторизация:** единый принцип токена. Веб — `ai_office_token` (localStorage). CLI — device-flow: `xerocode login` → код → токен, скоуп `organization_id` + воркспейс. MCP — тот же bearer как заголовок при вызове инструментов; IDE-расширение хранит токен в secret-storage редактора.
  - **Синхронизация состояния:** файлы/табы/diff/предложения — единое состояние воркспейса на бэке (per `organization_id`). Изменение из CLI (например, `xerocode apply`) отражается в вебе через polling/websocket; правка в IDE через мост поднимает diff в веб-ленту предложений.
  - **Источник предложения** виден в карточке (ПРЕДЛОЖЕНИЕ): мелкий бейдж `▸ CLI` / `▸ MCP` / `▸ IDE` / `▸ Web` (mono 9.5px, `--on-bg-dim`) — чтобы человек понимал, откуда пришёл diff.
  - **Все вызовы моделей — через прокси** (`use_proxy = bool(proxy)`), правило проекта №6: code-исполнение и генерация diff не обходят прокси-слой.

| **Конфликт** | Diff не применяется (файл изменился под мостом IDE / другим seat). Баннер `--xero-mute` + `lucide-react` `AlertTriangle`: «Файл изменился — обновите diff». Кнопки «Перечитать» (`RefreshCw`) / «Применить поверх». |
| **Suggestion: принят** | Карточка → тост «Применено» (`Check`, `--ws-code`), грязный-индикатор гаснет, diff закрыт, появляется «Отменить» (`Undo2`) на ~5с. |
| **Suggestion: отклонён** | Карточка fade-out ≤180ms; помечается dismissed; (ПРЕДЛОЖЕНИЕ) тонкая строка «Предложение отклонено · Вернуть» (`Undo2`, `--on-bg-dim`). |
| **AI думает (стрим предложения)** | Карточка-скелет: аватар «AI» + строки-плейсхолдеры (shimmer на `--xero-mute`), текст «AI анализирует код…» mono `--on-bg-mute`. Без спиннеров дольше 200ms — используем skeleton/стрим. |
| **Файл read-only** | Бейдж `lucide-react` `Lock` у имени файла (`--on-bg-dim`), «Только чтение». Apply скрыт/disabled, Run/Diff доступны. |

## Адаптив

- **≥1280px (десктоп):** полный `app-layout` 220px + 1fr. Toolbar в один ряд (имя слева, 3 кнопки справа). code-frame на всю ширину main. AI-suggestion полноразмерная под frame. Если включён run-output — он делит вертикаль с suggestion (табы или split).
- **768–1279px (планшет):** `app-side` сворачивается в узкую полосу-иконки (icon-rail ~64px) или прячется за бургер (`lucide-react` `Menu`). code-frame — на всю ширину. **Табы скроллятся горизонтально** при переполнении (overflow-x: auto, скрытый скроллбар, fade-маски по краям). Кнопки toolbar: при нехватке места `+ Apply changes` остаётся текстом+иконкой, `Run`/`Diff` могут стать icon-only (тултип по hover). AI-suggestion — снизу, на всю ширину.
- **<768px (телефон) — read-focused:** редактирование не основной сценарий (правило брифа: «code на телефоне — read-focused»).
  - `app-side` → bottom-nav или drawer.
  - code-body: горизонтальный скролл строк (никакого принудительного wrap кода, чтобы не ломать отступы Python); нумерация может сжиматься/прятаться.
  - **Табы — горизонтальный скролл** (свайп).
  - **AI-suggestion — снизу**, прижата к низу экрана (sticky bottom-sheet, как ключевой CTA): «Применить» крупной кнопкой во всю ширину, «Diff»/«Отклонить» вторичны. Это сохраняет паттерн «предложил → Apply» даже на мобиле.
  - Run-вывод — в раскрывающуюся снизу панель/шит.
  - Размеры шрифта кода не уменьшать ниже ~12px (читаемость).

## Данные

- **Файлы / табы:** `[{ id, name, path, language, dirty: bool, readonly: bool, active: bool }]`. Активный = подсвечен в табах и в toolbar-`h2`.
- **Содержимое файла:** текст + язык (для грамматики подсветки) + ревизия/hash (для детекта конфликта). Нумерация строк — производная от контента.
- **Diff:** `[{ type: 'add'|'del'|'ctx', oldLine, newLine, content }]`; агрегаты `{ added, removed }`; привязка к suggestion-id и к файлу/ревизии.
- **AI-предложения:** `[{ id, title, body, category: 'security'|'perf'|'refactor'|'fix', diffRef, source: 'cli'|'mcp'|'ide'|'web', status: 'open'|'applied'|'dismissed', model }]`. В мокапе: title=«добавь rotation токенов», body про refresh-rotation, category≈security.
- **run-output:** `{ status: 'idle'|'running'|'ok'|'error', stdout[], stderr[], exitCode, durationMs }`.
- **Источник CLI/MCP:** канал, инициировавший предложение/прогон; токен/seat, от которого пришло; для аудита и отображения бейджа источника.
- **Контекст модели:** какая модель сгенерировала предложение (см. model-picker воркспейса; для Код — multi-language-способные модели). Все запросы через прокси.

## Edge cases

- **Очень длинный файл:** виртуализация строк (рендерить видимое окно); нумерация и подсветка — лениво по вьюпорту. Не грузить весь highlight синхронно. (ПРЕДЛОЖЕНИЕ) миникарта справа (`lucide-react` `Map`?) — опционально, чтобы не утяжелять.
- **Много табов:** горизонтальный скролл полосы табов + (ПРЕДЛОЖЕНИЕ) оверфлоу-меню `lucide-react` `ChevronDown` со списком открытых файлов и поиском; «закрыть остальные».
- **Бинарник / не-текст:** не пытаться подсветить. Плейсхолдер: `lucide-react` `FileX` или `Image`/`Binary`, «Предпросмотр недоступен (бинарный файл)» + размер/тип; Run/Diff/Apply — disabled.
- **Большой diff:** сворачивание неизменённых участков («… N строк без изменений», кнопка «Развернуть», `lucide-react` `ChevronsUpDown`); навигация по чанкам; ленивый рендер. Предупреждение, если diff трогает много файлов (PR-helper-сценарий).
- **Долгий Run:** после порога — это уже не спиннер, а прогресс/стрим (правило ≤200ms спиннер). Кнопка «Стоп» (`Square`) всегда доступна; таймаут sandbox с понятным сообщением; стрим логов, чтобы не выглядело зависшим.
- **Конфликт ревизий (multi-seat / IDE-мост):** см. состояние «Конфликт» — diff на устаревшей ревизии не применять, форсить refresh.
- **Очень длинные строки:** горизонтальный скролл `code-content` (а не wrap, чтобы не ломать визуальную структуру кода); опциональный toggle «перенос строк» (`lucide-react` `WrapText`).

## Маппинг на код

**Каркас:** `app-layout` / `app-side` / `app-main` / `app-toolbar` / `btn-mini` — переиспользуем существующие классы из мастера (намespace `.x3-root` в `xerocode-v3.css`). В проде V2→V3 — это общий shell приложения (тот же, что A02/A04/A07/A08/A12).

**Новые компоненты (декомпозиция мокапа):**
- **`CodeEditor` / `CodeViewer`** — подсветка + нумерация.
  - **ПРЕДЛОЖЕНИЕ по движку:**
    - **CodeMirror 6** — рекомендуется. Легче Monaco, отлично для **read-focused** просмотра и подсветки, хорошо темизируется под CSS-переменные (тема пишется на наши токены), нормально работает на мобиле, multi-language через `@codemirror/lang-*` и Lezer-грамматики. Подходит, т.к. тяжёлое редактирование уходит в IDE пользователя.
    - **Monaco** — если нужен «VS Code в браузере» (полноценное редактирование, IntelliSense). Тяжелее, хуже на мобиле, своя система тем (придётся прокидывать токены в `defineTheme`). Брать, только если веб станет полноценным редактором (см. открытый вопрос).
    - **Рекомендация:** CodeMirror 6 для веб-экрана (просмотр + приём diff), Monaco-класс — только внутри IDE-моста, если потребуется.
  - В обоих случаях тему привязать к семантическим токенам подсветки (`--syntax-*`), НЕ к hardcoded hex.
- **`DiffView`** — unified/side-by-side. CodeMirror merge-аддон (`@codemirror/merge`) либо `react-diff-view`/`diff-match-patch` для рендера. Цвета — green/pink tint из палитры воркспейсов.
- **`SuggestionCard`** — карточка AI-предложения (фон `--xero-mute`, border `--xero`, аватар AI, заголовок `--display`, действия `btn-mini`). Стек/навигация для N предложений.
- **`RunOutput`** (ПРЕДЛОЖЕНИЕ — нет в мокапе) — моно-консоль вывода, стрим, exit-бейдж, кликабельные `file:line`.
- **`CodeTabs`** — полоса вкладок: active=`--bg`, скролл при переполнении, close/`●`-индикаторы, «+».

**CSS-переменные (используемые):** `--bg`, `--bg-soft`, `--bg-tile` (для вторичных панелей), `--on-bg`, `--on-bg-mute`, `--on-bg-dim`, `--line-bg`, `--line-bg-strong`, `--xero`, `--xero-mute`, `--xero-soft` (аватар-градиент), `--cream` (текст на primary), `--ws-code` (зелёный воркспейса — точечно: иконка/дот/строки/добавления diff/exit-ok), `--ws-text`/`--ws-video` (в палитре подсветки), `--display`, `--body`, `--mono`, `--ease`. Радиусы: 6 (btn-mini, табы), 10 (code-frame, suggestion-card), 999 (pills, если появятся).
**Новые семантические токены (ПРЕДЛОЖЕНИЕ):** `--syntax-keyword/fn/string/number/comment/base`, `--diff-add-bg/--diff-del-bg`, `--run-ok/--run-err`.

**lucide-react иконки (сводно):**
- Toolbar: `Play` (Run), `GitCompare` (Diff), `Check` (Apply).
- Сайдбар-пункт: `Code2` / `SquareTerminal`.
- Табы: `X`, `Plus`, `FileCode`.
- Suggestion: `Check`, `GitCompare`, `X`, `RotateCcw`, `Copy`, `ShieldCheck`/`Zap`/`Sparkles`/`Bug` (категории), `ChevronLeft`/`ChevronRight` (навигация по предложениям).
- Diff: `Rows`/`Columns` (режим), `ChevronUp`/`ChevronDown`, `ChevronsUpDown` (сворачивание).
- Run-output: `Loader2` (spin), `Square` (Stop), `RefreshCw`.
- Состояния/edge: `FileCode2`/`FileX`/`Lock`/`AlertTriangle`/`Undo2`/`WrapText`/`Menu`.
- **Запрет:** никаких emoji-иконок; все глифы `▸`/`+` в проде → lucide или mono-лейбл.

## Открытые вопросы / ПРЕДЛОЖЕНИЯ

1. **Редактирование vs read-only (ключевая развилка).**
   - **ПРЕДЛОЖЕНИЕ:** веб-экран — **read-focused + приём предложений** (просмотр, Run, Diff, Apply, dismiss), реальное редактирование — в IDE пользователя через мост/CLI. Это совпадает с правилом «code на телефоне — read-focused» и оправдывает выбор CodeMirror вместо Monaco. Полноценный браузерный редактор — отдельное решение/итерация.
2. **Git-интеграция.**
   - **ПРЕДЛОЖЕНИЕ:** Apply = правка рабочей копии; отдельный слой git — `Commit` / `Создать PR` (`GitPullRequest`), статус ветки, список изменённых файлов (PR/MR-helper из спеки воркспейса). Граница Apply↔commit должна быть явной в UI.
3. **Терминал / run-output.**
   - **ПРЕДЛОЖЕНИЕ:** run-output как панель/таб (нет в мокапе, но Run-кнопка есть → выводу нужно место). Полный встроенный терминал (xterm.js) — отдельный вопрос; CLI как продукт уже закрывает консольный сценарий вне веба.
4. **Подсветка: hardcoded hex → токены.** `.s #10B981` и `.n #F59E0B` нарушают правило цветовых переменных. **ПРЕДЛОЖЕНИЕ:** ввести `--syntax-*` (маппинг выше) при декомпозиции; заодно подготовить light-вариант палитры.
5. **Источник предложения в карточке (CLI/MCP/IDE/Web).** Показывать бейдж канала — да/нет? **ПРЕДЛОЖЕНИЕ:** да, mono-бейдж — для прозрачности и аудита в команде.
6. **Множественные предложения.** Стек карточек vs «1 из N» с навигацией? **ПРЕДЛОЖЕНИЕ:** «1 из N» + ChevronLeft/Right, чтобы не разрастался экран.
7. **Иконка воркспейса в сайдбаре.** Сейчас generic `.ico`. **ПРЕДЛОЖЕНИЕ:** `Code2` с дотом `--ws-code`; active-строка остаётся `--xero-mute` (единый язык навигации) — подтвердить, что green допускается только как дот/иконка, не как фон строки.
8. **Multi-seat одновременная работа над файлом.** Нужны ли presence-индикаторы (кто открыл файл)? **ПРЕДЛОЖЕНИЕ:** на старте — детект конфликта по ревизии (см. состояние «Конфликт»); полноценное co-editing — позже.
9. **Подключение репозитория.** Откуда берутся файлы в вебе: загрузка / clone / IDE-мост / CLF push? **ПРЕДЛОЖЕНИЕ:** в пустом состоянии — «Подключить через CLI» (device-flow) как основной путь, плюс «Открыть файл» для разовых.
